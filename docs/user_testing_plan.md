# Paper Agent 用户试用与 MVP 验证计划

## 1. 验证目标

Paper Agent 当前要验证的不是“能不能做一个完整 Zotero 替代品”，而是一个更小、更真实的闭环：

```text
导入 10-30 篇真实 PDF
  -> 自动抽取全文并建库
  -> 用户用自然语言提问
  -> AI 回答带论文出处和原文片段
  -> 用户点击出处回到 PDF 对应页
```

核心假设：

> 处在开题、综述、毕设、论文 introduction 阶段的用户，愿意把自己的真实论文 PDF 导入 Paper Agent，并因为“回答可追溯到原文”持续使用它完成阅读和整理任务。

本轮验证不以用户口头称赞为成功标准，只看真实行为。

## 2. 早期用户画像

| 编号 | 用户画像 | 典型任务 | 当前替代方案 | 为什么适合早期验证 |
|---|---|---|---|---|
| U1 | 研一/研二刚开始开题的学生 | 读 20-50 篇方向相关论文，整理研究问题 | Zotero + ChatGPT/Kimi 单篇上传 + Word 摘录 | 痛点强，流程混乱，愿意试新工具 |
| U2 | 正在写文献综述的研究生 | 找不同论文对同一概念的定义、方法和不足 | 手动搜索 PDF、复制摘要、问大模型 | 高频需要跨论文对比 |
| U3 | 本科毕设学生 | 快速理解导师给的 10-30 篇参考论文 | 浏览器 PDF 阅读 + 大模型网页问答 | 时间紧，愿意用省时间工具 |
| U4 | 写论文 introduction 的研究生 | 找领域背景、已有方法、研究空白 | Zotero 标签 + 手动摘录引用 | 强依赖出处和原文定位 |
| U5 | 准备组会汇报的学生 | 一两天内读完几篇论文并做 PPT | PDF 阅读器 + 手写笔记 | 有明确截止时间，验证 urgency |
| U6 | 跨方向入门的科研新人 | 快速搞清领域核心术语和论文关系 | 搜索引擎 + 综述论文 + AI 聊天 | 对“本地论文库问答”价值感知强 |
| U7 | 导师/课题组助理 | 帮课题组整理某方向论文资料 | 文件夹 + Excel + Zotero 群组库 | 可能影响多人使用，但早期先观察个人流程 |
| U8 | 需要读英文论文的中文用户 | 用中文问英文 PDF 内容并回看原文 | 翻译软件 + 单篇 PDF AI 工具 | 对中文问答和英文出处并存需求明显 |
| U9 | 内网/学校数据库重度用户 | 已经有合法下载的 CNKI/IEEE/ScienceDirect PDF | 学校库下载 + 本地文件夹 | “爬不到内网也能导入本地 PDF”是差异点 |
| U10 | AI 工具敏感但担心幻觉的用户 | 想用 AI 提速，但必须确认答案来自原文 | ChatGPT + 人工核对 | 最能测试出处和可信度是否成立 |

优先招募顺序：U1、U2、U3、U4、U5。不要一开始找“泛科研人员”，要找手上已经有论文任务的人。

## 3. 30 分钟试用脚本

### 试用前准备

- 参与者自带 10-30 篇真实 PDF，最好来自当前正在做的开题、综述、毕设或论文写作任务。
- 允许用户使用自己原本的工具作为对照，例如 Zotero、浏览器 PDF、ChatGPT、Kimi、Word、Excel。
- 观察者只记录行为，不解释产品设计意图。
- 试用时开启录屏或由观察者手动记录关键节点。

### 0-3 分钟：背景确认

让用户口头回答：

- 你最近一次集中读论文是什么时候？
- 这批论文是为了开题、综述、毕设、组会，还是写论文？
- 你现在用什么工具管理和阅读这些 PDF？
- 上一次你想从一堆论文里找答案时，是怎么做的？

观察重点：

- 用户是否真的处在论文压力场景。
- 用户当前是否已有低效替代流程。
- 用户是否带来了真实 PDF，而不是随便找几篇测试文件。

### 3-8 分钟：导入 PDF

任务：

1. 用户打开 Paper Agent。
2. 用户导入 10-30 篇 PDF。
3. 用户查看系统是否显示导入结果。

观察重点：

- 用户是否能自己找到导入入口。
- 导入过程是否卡住、失败或没有反馈。
- 用户是否能理解“解析成功、页数、chunk 数、建库状态”。
- 用户是否会问“我导入的论文在哪里”。

记录指标：

- 导入 PDF 数量。
- 成功解析数量。
- 用户第一次遇到困惑的位置。
- 从打开产品到完成第一篇导入的时间。

### 8-13 分钟：找到刚导入的论文并打开 PDF

任务：

1. 用户找到刚导入的论文列表。
2. 用户打开其中一篇 PDF。
3. 用户翻页、缩放，并确认这是自己导入的论文。

观察重点：

- 用户是否能从“导入完成”自然进入“阅读 PDF”。
- PDF 查看器是否稳定、清晰、不卡顿。
- 用户是否需要通过搜索才能找到刚导入的论文。
- 用户是否期待最近导入、论文库、项目空间等入口。

记录指标：

- 用户是否无需提示就能打开 PDF。
- 从导入完成到打开 PDF 的时间。
- 是否出现“找不到论文”的行为。

### 13-22 分钟：基于本地论文提问

给用户三个任务，但不要替用户组织问题：

1. 问一个论文理解问题：例如“这篇论文主要解决什么问题？”
2. 问一个跨论文比较问题：例如“这些论文里哪些方法都用了 RAG？”
3. 问一个写作辅助问题：例如“哪些论文可以支持我 introduction 里的研究背景？”

观察重点：

- 用户是否自然提出第二个、第三个问题。
- 用户是否追问“依据是什么”。
- 用户是否会点击来源回看原文。
- 用户是否相信回答，还是立刻转回原工具核对。

记录指标：

- 10 分钟内提问次数。
- 有多少问题基于真实论文任务。
- AI 回答是否带来源。
- 用户是否点击来源。
- 用户是否认为来源片段足以支撑答案。

### 22-27 分钟：引用与原文核对

任务：

1. 用户从 AI 回答中选一个关键结论。
2. 用户点击引用或来源。
3. 用户在 PDF 中核对原文。
4. 用户判断这个回答是否能用于自己的笔记或写作。

观察重点：

- 引用是否有论文名、页码、原文片段。
- 点击后是否能跳转到对应 PDF 页。
- 用户是否能快速确认答案没有胡编。
- 如果跳转不准，用户是否还愿意继续用。

记录指标：

- 点击引用次数。
- 成功跳转到 PDF 页次数。
- 用户主观可信度评分，1-5 分。
- 用户是否把回答复制到自己的笔记/文档。

### 27-30 分钟：结束访谈

只问过去行为和下一步行为，不问“你觉得好不好”：

- 你刚才有没有导入真实项目里的论文？
- 这 30 分钟里，有没有一个回答是你想保存到笔记里的？
- 如果明天继续写综述/开题，你会不会重新打开它？为什么？
- 哪一步让你最不放心？
- 你现在还会继续用哪个原来的工具？为什么？

## 4. 通过/失败标准

### MVP 通过标准

一次试用算通过，必须同时满足：

- 用户导入至少 10 篇真实 PDF。
- 用户能在 5 分钟内找到导入结果并打开 PDF。
- 用户在 10 分钟内提出至少 3 个真实研究问题。
- 用户至少点击 2 次来源或引用回看原文。
- 用户对至少 1 条回答给出 4/5 或以上可信度评分。
- 用户表示第二天愿意继续用它处理同一批论文，并在 24 小时内真的回来使用一次。

### 强信号

- 用户主动导入更多 PDF，而不是只完成任务。
- 用户把 AI 回答复制到自己的笔记、Word、PPT 或 Zotero notes。
- 用户问“能不能按项目/课题保存这一批论文”。
- 用户问“能不能给我同学/组员也用”。
- 用户发现回答不够好时，仍愿意通过换问法继续探索。

### 弱信号

- 用户说“挺好”，但不导入真实 PDF。
- 用户只问一个泛泛问题就停止。
- 用户不点击来源，也不核对原文。
- 用户把它当成单篇 PDF 聊天工具，而不是自己的论文库。
- 用户第二天没有回来，也没有任何主动反馈。

### 失败标准

出现以下任一情况，本轮 MVP 假设应判定为失败或需要 pivot：

- 5 个目标用户中少于 2 个愿意导入真实 PDF。
- 用户普遍找不到导入后的论文。
- 用户不信任 AI 回答，即使回答带了来源。
- 引用无法回到原文页，导致用户认为不如直接用 PDF 阅读器。
- 用户认为现有工具组合已经足够，不愿意改变流程。

## 5. 试用前必须补齐的功能清单

### P0：没有这些不应该找用户试

| 功能 | 验收标准 | 原因 |
|---|---|---|
| 最近导入论文列表 | 上传后能在首页或文献库看到最近导入的论文 | 用户不能靠搜索找自己刚上传的东西 |
| PDF 导入状态反馈 | 显示成功/失败、页数、chunk 数、解析状态 | 用户需要知道系统是否真的建库 |
| 一键打开 PDF 查看器 | 每篇本地 PDF 都有明显入口 | 核心闭环必须能从论文回到原文 |
| AI 回答附来源 | 每条基于论文的回答至少显示论文名、页码、片段 | 学术场景里可信度比流畅度重要 |
| 无依据时拒答 | 本地库检索不到时明确说明，不编造 | 防止第一次试用就失去信任 |
| 基础错误提示 | PDF 解析失败、接口失败、模型失败时有可读提示 | 试用现场不能只显示空白或转圈 |

### P1：最好在第一轮 5-10 人前补齐

| 功能 | 验收标准 | 原因 |
|---|---|---|
| 点击引用跳转 PDF 页码 | 来源中的页码能打开 PDF 并定位到页 | 这是从“AI 聊天”变成“可信研究工具”的关键 |
| 论文详情页 | 显示标题、作者、摘要、页数、chunk 数、导入时间 | 用户需要确认系统理解了自己的论文 |
| 问答记录保留 | 刷新后仍能看到本轮对话 | 用户会把它当笔记流使用 |
| 检索调试信息 | 至少开发者能看到命中的 chunk 和分数 | 方便定位“答错是检索错还是模型错” |
| 批量上传体验 | 10-30 篇 PDF 上传不会让用户逐个痛苦操作 | 验证目标就是一批论文，而不是单篇 |

### P2：验证通过后再做

| 功能 | 验收标准 | 原因 |
|---|---|---|
| 文献图谱 | 能展示论文引用或主题关系 | 价值不错，但不是第一假设 |
| 自动爬开放 PDF | 能从 arXiv 等开放来源下载并导入 | 加分项，不应替代用户真实 PDF 验证 |
| 标签/项目空间 | 用户可按课题管理论文集 | 等用户真的有多批论文后再做 |
| 引文格式导出 | 支持 BibTeX/GB/T/APA 等 | 写作后段需求，不是首轮关键 |
| 成本看板 | 展示每次问答 token 和成本 | 对长期运营重要，但首轮可手工记录 |

## 6. 第一轮试用安排

### 招募目标

- 第 1 轮：5 人。
- 第 2 轮：补到 10 人。
- 每人必须带真实论文任务和真实 PDF。
- 优先找同校、同专业、正在写开题/综述/毕设的人。

### 观察记录表

| 字段 | 记录方式 |
|---|---|
| 用户编号 | U1-U10 |
| 场景 | 开题/综述/毕设/组会/introduction |
| PDF 数量 | 实际导入数量 |
| 成功解析数量 | 系统返回 |
| 第一次卡住的位置 | 观察记录 |
| 30 分钟提问次数 | 数字 |
| 点击引用次数 | 数字 |
| 是否复制回答到笔记 | 是/否 |
| 可信度评分 | 1-5 |
| 明天是否回来 | 是/否 |
| 24 小时后是否真的回来 | 是/否 |
| 最大阻碍 | 用户原话 |

### 结果判定

- 如果 5 人中有 3 人达成 MVP 通过标准：继续打磨这个方向。
- 如果 5 人中只有 1-2 人通过：缩小 ICP，优先服务最痛的场景，例如“开题/综述前两周”。
- 如果 5 人中 0 人通过：停止加功能，重新访谈当前论文阅读流程，确认是否问题定义错了。

## 7. 本轮不要做的事

- 不要把产品包装成 Zotero 替代品。
- 不要优先做大规模爬虫。
- 不要为了演示效果导入假论文。
- 不要用“用户说愿意用”当作成功。
- 不要在引用和 PDF 回跳没做好前强调 AI 很强。
- 不要找没有真实论文任务的人试用。

## 8. 下一步行动

1. 先补齐 P0 功能，尤其是最近导入论文列表、导入状态反馈、AI 回答来源。
2. 用开发者自己的 10-30 篇 PDF 跑一遍完整 30 分钟脚本。
3. 找 2 个熟人做预试用，修掉明显卡点。
4. 再找 5 个目标用户做正式试用。
5. 根据行为数据决定继续做 Paper Agent，还是缩小成“综述写作 PDF 知识库助手”。
