# Paper Agent 问卷调研结论与 MVP 路线

## 1. 样本质量判断

本轮问卷目前应视为“有效但样本量很小”的定性样本。它不能证明市场规模，也不能直接代表所有科研用户，但它对 MVP 方向有足够强的信号：用户不是在泛泛地说“想要 AI 读论文”，而是在描述一个具体任务链路。

这份样本的质量较高，原因是：

- 场景明确：开题前快速了解方向、组会/PPT 汇报、课程/期末论文。
- 任务明确：围绕“中国道路史”等主题查找资料、论文依据和出处。
- 替代方案明确：ChatGPT/Kimi/豆包单篇上传，加浏览器搜索。
- 痛点明确：AI 回答没有出处，不敢直接相信；摘录和引用耗时。
- MVP 选择明确：用户单选最想要的是“AI 问答带页码和原文出处”。

限制也必须承认：

- 目前更像单个强信号样本，不足以判断付费意愿。
- 样本偏文史/思政/课程论文场景，不一定覆盖理工科实验论文。
- 用户提到权威资料源，说明 Paper Agent 的“论文库”可能需要扩展成“证据资料库”，但这需要单独验证。

当前结论：可以用它指导下一阶段研发，但不能用它宣布 PMF。

## 2. 核心场景

### 场景 A：开题前快速了解方向

用户需要在短时间内知道一个主题有哪些核心概念、代表观点、已有研究和可引用来源。此时最重要的不是“自动写开题报告”，而是快速找到可靠出处。

典型问题：

- “中国道路史这个主题有哪些常见研究角度？”
- “哪些论文或资料能支持这个论点？”
- “这个结论具体出自哪里？”

### 场景 B：组会/PPT 汇报

用户需要把资料整理成可以展示的内容，尤其需要每个观点背后有可核对来源。AI 如果只给一段流畅总结，但没有论文名、页码和原文片段，对汇报帮助有限。

典型问题：

- “这几篇资料里对某个历史阶段的说法有什么共同点？”
- “帮我找 3 条可以放进 PPT 的观点，每条都要带出处。”
- “这段话能不能回到原文？”

### 场景 C：课程/期末论文

用户需要围绕一个主题查资料、摘录、引用，并形成论证素材。当前做法往往是浏览器搜索加 AI 问答，但 AI 的出处不稳定，导致用户仍要回头手动查。

典型问题：

- “这个主题有哪些可引用资料？”
- “哪篇文章提到过这个结论？”
- “帮我找原文中可以引用的一段。”

## 3. 最强痛点

最强痛点不是“没有 AI”，而是：

> AI 给了答案，但用户不知道答案来自哪篇论文、哪一页、哪段原文，所以不敢用。

痛点排序：

| 排名 | 痛点 | 强度 | 为什么重要 |
|---:|---|---|---|
| 1 | 找不到结论出处 | 高 | 学术写作和汇报不能只靠 AI 口头总结 |
| 2 | AI 回答不带页码和原文 | 高 | 用户无法快速查证，信任成本高 |
| 3 | 摘录和引用耗时 | 高 | 用户真正想节省的是从资料到证据的整理时间 |
| 4 | 单篇上传工具割裂 | 中 | ChatGPT/Kimi/豆包可用，但资料库无法沉淀 |
| 5 | 浏览器搜索结果分散 | 中 | 搜得到资料，但很难直接转成可用证据 |

这说明 MVP 的核心不是做“更会聊天的 AI”，而是做“能帮用户找证据的 AI”。

## 4. 研发路线判断

第一阶段必须聚焦：

> 找出处 / 查证据 / 回原文。

不要把第一阶段做成：

- Zotero 替代品。
- 自动写完整论文工具。
- 大而全的文献图谱。
- 精美笔记系统。
- 多人协作资料平台。

正确的一句话定位：

> Paper Agent 第一版是一个“证据型论文/资料问答工具”：用户导入或接入可信资料后，AI 的每个关键回答都必须能回到原文页码和片段。

第一阶段成功标准：

- 用户能导入或接入资料。
- 用户能问一个真实论文/汇报问题。
- 系统能返回观点、来源标题、页码、原文片段。
- 用户能点击来源回到 PDF 或资料页。
- 用户愿意把回答摘录到自己的课程论文、PPT 或开题笔记里。

## 5. 功能优先级

### P0：第一阶段必须做

| 功能 | 验收标准 | 对应痛点 |
|---|---|---|
| 证据型 AI 回答 | 回答中每个关键结论至少绑定 1 个来源 | AI 回答不敢信 |
| 页码和原文片段 | 来源显示资料名、页码、原文摘录 | 找不到出处 |
| 点击回到原文页 | 点击引用后打开 PDF 对应页或资料原文位置 | 查证耗时 |
| 最近导入/资料入口 | 用户能看到刚导入的资料，不需要靠搜索找 | 资料库沉淀 |
| 无依据拒答 | 检索不到证据时明确说“当前资料库未找到依据” | 防止幻觉 |
| 基础资料类型标识 | 区分论文、官方史料、讲话、网页、其他资料 | 权威资料源需求 |

### P1：第一轮用户试用前尽量做

| 功能 | 验收标准 | 为什么排 P1 |
|---|---|---|
| 批量导入 PDF | 一次导入多篇资料并显示解析结果 | 真实任务往往不是单篇 |
| 证据卡片 | 每条来源以卡片展示：标题、页码、片段、打开按钮 | 降低查证成本 |
| 资料搜索过滤 | 按主题、资料类型、来源筛选 | “中国道路史”等主题需要聚合 |
| 问答记录留存 | 用户能回看自己问过的问题和证据 | 汇报/论文写作需要反复整理 |
| 权威来源标签 | 对官方资料、党史资料、讲话等标注来源类型 | 文史/思政场景强需求 |

### P2：验证通过后再做

| 功能 | 验收标准 | 为什么暂后 |
|---|---|---|
| 文献图谱 | 能展示引用/主题关系 | 有价值，但不是本样本最强痛点 |
| 多人协作 | 小组共享资料库和问答 | 等个人闭环成立后再扩展 |
| 精美笔记系统 | 高级编辑、排版、卡片库 | 先证明证据查找有用 |
| 自动写完整论文 | 生成完整正文 | 风险高，且偏离“查证据” |
| 大规模自动爬虫 | 自动抓取大量资料 | 需要处理版权、权限和质量控制 |

## 6. 暂缓功能

以下功能第一版应明确砍掉或只保留入口，不投入主要研发：

- 复杂文献图谱：用户当前最急的是出处和页码，不是关系网络。
- 多人协作：先验证个人是否愿意持续用，再做团队。
- 自动写完整论文：容易变成低可信写作工具，且学术风险高。
- 精美笔记系统：用户已经有 Word/PPT/笔记软件，先让证据可复制即可。
- 泛化聊天助手：如果回答不能绑定出处，就不符合这轮样本需求。

## 7. 权威资料源方向

用户提到希望对接中央党史资料库、官方史料、领导人讲话等权威资料源，这是重要信号，但应谨慎处理。

第一阶段可以做：

- 允许用户导入合法获得的官方 PDF、网页打印 PDF、资料汇编。
- 在资料元数据里增加 `source_type`，例如 `paper`、`official_archive`、`speech`、`webpage`、`book_excerpt`。
- 在回答来源中显示资料类型，让用户知道答案来自论文还是官方资料。

暂时不要做：

- 绕过权限或登录的自动抓取。
- 对官方资料源做未经确认的大规模爬取。
- 把所有权威资料源混在一起，导致来源质量不可控。

下一步可验证：

- 用户是否真的拥有可导入的权威资料。
- 用户是否更信任官方资料源而不是普通论文。
- 哪些资料源对课程论文/开题最常用。

## 8. 下一轮问卷应该补问什么

下一轮不要问“你愿不愿意用”，要问过去行为和具体任务。

建议补问：

1. 你最近一次写课程论文/开题/组会 PPT 是什么时候？
2. 当时你用了多少篇资料或论文？
3. 你最后引用了多少条出处？
4. 哪一步最耗时：找资料、读资料、摘录、引用格式、核对原文，还是写作？
5. 你有没有因为 AI 没有出处而放弃使用某个回答？
6. 你现在如何保存可引用片段？Word、Excel、Zotero、浏览器收藏，还是手写？
7. 如果 AI 给出观点、资料名、页码、原文片段，你会把它复制到哪里？
8. 你常用哪些权威资料源？是否能合法下载 PDF 或网页资料？
9. 你是否愿意导入自己的真实资料给本地工具处理？担心什么？
10. 如果只保留一个功能，你会选：找出处、自动总结、文献图谱、笔记整理、还是自动写作？
11. 你是否愿意第二天继续使用同一个资料库，而不是重新上传给 ChatGPT/Kimi？
12. 你愿意为这个工具付费吗？如果愿意，是个人付费、课题组付费，还是学校/课程统一购买？

## 9. 给其他线程的协作建议

### 9.1 资料入口线程

目标：让用户能把论文和权威资料放进系统，并明确看到“资料已经可被检索”。

建议实现：

- 首页或文献页增加“最近导入资料”区域。
- 上传后显示标题、资料类型、页数、chunk 数、解析状态、导入时间。
- 支持基础 `source_type`：论文、官方史料、领导人讲话、网页资料、其他。
- 不要只叫“论文”，可以逐步改成“资料库”或“证据库”，但仍保留论文场景。
- 导入失败要给明确原因，例如 PDF 无文本层、解析失败、文件过大。

接口建议：

- 资料列表 API 返回 `id`、`title`、`source_type`、`file_path`、`parse_status`、`page_count`、`chunk_count`、`created_at`。
- 上传/导入 API 返回同样的摘要信息，方便前端马上刷新列表。

### 9.2 证据型回答线程

目标：AI 回答不再只是自然语言总结，而是每个关键结论都能绑定证据。

建议实现：

- Agent 调用 `search_chunks` 后，把命中的 chunk 作为结构化 sources 返回。
- 回答格式强制包含“结论 + 证据”。
- 每条证据至少包含 `paper_id`、`title`、`page_start`、`page_end`、`chunk_id`、`snippet`。
- 如果检索结果不足，回答必须明确说明“当前资料库没有找到足够依据”。
- 不要让模型自己编页码，页码必须来自 chunk metadata。

接口建议：

```json
{
  "answer": "...",
  "sources": [
    {
      "paper_id": "...",
      "title": "...",
      "source_type": "paper",
      "page_start": 12,
      "page_end": 13,
      "chunk_id": "...",
      "snippet": "..."
    }
  ]
}
```

### 9.3 PDF 跳页线程

目标：从 AI 来源点击后，用户能直接回到原文页码。

建议实现：

- PDF reader 支持通过 URL 或函数参数打开指定页，例如 `/reader?paper_id=...&page=12`。
- 来源卡片中的页码按钮调用 reader 并传入 `paper_id` 和 `page_start`。
- reader 打开后高亮当前页对应 chunk。
- 如果页码范围是 12-13，默认打开 `page_start`，并在侧栏提示范围。
- 对没有 PDF 的网页/官方资料，提供“打开原文 URL”或“查看文本片段”。

接口建议：

- `GET /api/papers/{paper_id}/pdf` 继续返回 PDF。
- `GET /api/papers/{paper_id}/chunks` 需要稳定返回 `page_start`、`page_end`、`content`。
- 前端 reader 状态需要能接收 `paperId` 和 `page`。

## 10. 研发路线

### 第一阶段：找出处 / 查证据

时间目标：1-2 周。

只做一个闭环：

```text
导入资料
  -> 自动解析和切块
  -> 用户提问
  -> AI 给回答
  -> 回答显示证据卡片
  -> 点击证据回到 PDF 页
```

验收标准：

- 5 个目标用户中至少 3 个愿意导入真实资料。
- 每个用户至少提出 3 个真实问题。
- 每个用户至少点击 2 次证据卡片。
- 至少 2 个用户把回答或原文片段复制到自己的论文/PPT/笔记里。

### 第二阶段：资料源扩展

在第一阶段通过后，再做权威资料源和资料类型扩展。

重点：

- 官方史料和讲话资料的导入。
- 网页资料保存为可引用文本。
- 来源权威性标注。
- 按主题构建小型资料包。

### 第三阶段：写作辅助

只有在证据闭环稳定后，才做写作辅助。

重点：

- 根据证据生成段落草稿。
- 引用格式辅助。
- PPT 观点卡片。
- 综述/开题材料整理。

仍然不要一开始承诺自动写完整论文。

## 11. 当前产品定位更新

旧定位容易变宽：

> 本地论文记忆库和 RAG 学术助手。

当前更准确的 MVP 定位：

> Paper Agent 是一个面向开题、组会和课程论文的证据型资料问答工具。它帮助用户从自己的论文和权威资料中查找可核对出处，让 AI 回答能回到原文页码。

一句话原则：

> 第一版只追求“回答可查证”，不追求“功能很全”。
