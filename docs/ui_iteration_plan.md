# Paper Agent 本轮 UI 迭代计划与验收标准

## 1. 迭代定位

本轮 UI 需求不是新的产品方向，也不是把 Paper Agent 扩成 Zotero 替代品、翻译软件或自动写论文工具。它的目标是让第一阶段 MVP 更适合真实用户试用：

```text
导入 PDF
  -> 在阅读器里读原文
  -> 问 AI / 找出处
  -> 回答结构清楚、可滚动、可核对
  -> 点击来源回到 PDF 页
  -> 必要时做中英/中文对照理解
```

本轮 UI 改动应服务于一个核心判断：

> 用户是否因为“答案清楚、出处可点、原文能查”而愿意把 Paper Agent 放进开题、组会/PPT、课程论文的资料工作流。

## 2. 本轮需求拆解

| 需求 | 用户价值 | MVP 关系 |
|---|---|---|
| AI 回答 Markdown 美化 | 让回答更像正式资料卡片，便于阅读和复制 | 增强证据型回答的可信度 |
| AI 逐字生成，可自由滚动 | 降低等待焦虑，保留网页版 AI 的交互感 | 提升试用体验，不改变能力边界 |
| 阅读器隐藏 chunk 列表 | 减少干扰，让用户专注 PDF 原文 | chunk 仍是检索基础，但不直接暴露给用户 |
| 阅读器双栏对照翻译器 | 帮用户理解英文/难读原文 | 辅助阅读，不替代证据链 |
| 阅读器内查出处 | 用户阅读时可直接验证观点来源 | 强化“找出处 / 查证据 / 回原文”闭环 |

## 3. AI Markdown 美化验收标准

### 3.1 显示内容

AI 回答应支持并正确渲染：

- 段落。
- 加粗。
- 有序列表。
- 无序列表。
- 行内代码或术语。
- 引用块。
- 简短代码块。
- 表格，至少在桌面端可横向阅读。
- 来源卡片或“本地引用来源”区域。

证据型回答建议结构：

```markdown
结论：
...

依据：
1. 《论文/资料名》，p.12：原文片段...
2. 《论文/资料名》，p.15：原文片段...

可用性判断：
这条证据可以/不可以直接用于论文或 PPT，原因是...
```

### 3.2 什么算好

通过标准：

- 用户一眼能区分“结论”和“依据”。
- 来源标题、页码、原文片段视觉上比普通正文更容易定位。
- 长回答不会挤成一大坨纯文本。
- 列表、表格、引用块在聊天窗口内不溢出、不遮挡按钮。
- 复制回答到 Word/PPT/笔记时，基本结构仍可读。
- 没有 Markdown 原始符号泄露，例如 `**标题**` 没被渲染。

失败表现：

- 回答仍是纯文本大段落。
- 来源和正文混在一起，用户找不到页码。
- Markdown 渲染导致 XSS 风险或 HTML 被直接执行。
- 表格撑破聊天窗口。
- 移动或窄屏时文字互相遮挡。

### 3.3 暂不支持范围

第一版不要求：

- 复杂数学公式渲染。
- Mermaid 图。
- 复杂嵌套表格。
- 富文本编辑器式排版。
- 自动生成论文格式参考文献。

## 4. 打字机输出与滚动逻辑验收标准

### 4.1 交互目标

AI 输出应像常见网页版 AI 一样逐字或逐段出现，让用户感到系统正在工作。同时，用户必须可以自由滚动，不被强制拉到底部。

### 4.2 通过标准

打字机输出：

- AI 开始响应后，先出现 loading 或空 assistant 消息气泡。
- 内容以稳定节奏追加，不要求真正 token streaming，但视觉上应逐字/逐段生成。
- Markdown 在生成过程中不应频繁破版。
- 生成结束后，最终 Markdown 完整渲染。
- 如果接口失败，已显示的内容不应消失，并给出错误提示。

滚动逻辑：

- 如果用户当前在聊天底部，生成时自动跟随最新内容。
- 如果用户手动向上滚动查看历史，系统停止强制自动滚到底。
- 用户回到底部后，继续自动跟随。
- 长回答生成时，输入框和发送按钮不能被内容覆盖。
- 移动端或窄屏下，聊天窗口仍可独立滚动。

性能标准：

- 长回答生成时页面不卡死。
- 不因为每个字符都重渲染整个聊天列表导致明显掉帧。
- 关闭/打开 AI 二级界面后，当前消息状态不丢失。

### 4.3 失败表现

- 用户往上滚动时被反复拉回底部。
- 正在输出时页面无法滚动。
- Markdown 每生成几个字就闪烁。
- 生成到一半切换页面后消息丢失。
- 生成失败后只剩空白气泡，没有错误提示。

## 5. 阅读器隐藏 Chunk 列表验收标准

### 5.1 目标

chunk 列表是系统内部检索结构，不应该作为第一版阅读器的主 UI 暴露给真实用户。本轮应隐藏或弱化 chunk 列表，让阅读器更像正常 PDF 阅读工具。

### 5.2 隐藏后不能坏的功能

即使隐藏 chunk 列表，也必须保留：

- PDF 打开。
- 上一页 / 下一页。
- 页码输入跳转。
- 缩放。
- 从 AI 来源或找出处结果跳到 `page_start`。
- 当前来源页码的定位能力。
- 阅读器返回上一视图。
- 阅读器内查出处入口。

### 5.3 通过标准

- 默认界面不显示原始 chunk 列表。
- 用户从来源卡片打开 PDF 后，能直接到对应页。
- 如果需要高亮来源片段，使用轻量提示或来源摘要，不恢复长 chunk 列表。
- 开发调试信息不出现在普通用户界面。
- 没有 chunk 列表后，阅读器视觉更清爽，不影响 PDF 可读区域。

### 5.4 可保留的开发模式

可以保留一个仅开发者使用的调试入口，例如：

- URL 参数 `?debug_chunks=1`。
- 隐藏按钮。
- 控制台调试。

但第一轮用户试用时不要主动展示。

## 6. 阅读器双栏对照翻译器验收标准

### 6.1 第一版支持范围

第一版翻译器只做阅读辅助，不做学术翻译产品。

建议支持：

- 用户选中 PDF 中一段文本后，点击“翻译”。
- 或输入/粘贴一段原文后翻译。
- 左栏显示原文。
- 右栏显示中文翻译。
- 支持英文到中文。
- 支持中文资料的“白话解释/改写”，但需要明确标注为解释，不是原文。
- 保留来源页码，例如“来自 p.12”。

### 6.2 双栏布局通过标准

- 左右两栏在桌面端并排显示。
- 窄屏下可上下堆叠或通过 tabs 切换。
- 左栏原文不可被翻译覆盖。
- 右栏翻译可复制。
- 翻译结果和当前 PDF 页码关联。
- 切换 PDF 页时，不应误以为旧翻译来自新页面。

### 6.3 翻译质量边界

第一版必须明确：

- 翻译只帮助理解，不作为可引用原文。
- 引用时仍应回到 PDF 原文。
- 如果翻译失败，保留原文，不清空用户选择。

### 6.4 暂不支持范围

第一版不做：

- 整篇 PDF 自动全文翻译。
- 保持论文原版排版的双语 PDF。
- 批量翻译多篇论文。
- 专业术语词库管理。
- 文献级翻译记忆库。
- 自动把翻译写入论文。

## 7. 阅读器内查出处验收标准

### 7.1 目标

用户在读 PDF 时，可以直接问“这段话/这个观点有没有本地依据”，不必回到外部搜索页或聊天窗口。

### 7.2 第一版交互

建议入口：

- 阅读器顶部或侧边提供“查出处”输入框。
- 用户可以输入一句观点。
- 如果用户选中了 PDF 文本，可以一键把选中文本填入查出处框。
- 返回结果以证据卡片形式展示，而不是原始 chunk 列表。

证据卡片至少包含：

- 资料标题。
- 页码范围。
- 原文片段。
- 检索方式或置信提示。
- “打开原文页”按钮。

### 7.3 通过标准

- 在阅读器内输入问题后，不离开 PDF 页面即可看到来源结果。
- 点击结果能打开对应 PDF 页。
- 如果结果来自当前 PDF，保持当前阅读器上下文。
- 如果结果来自其他 PDF，明确提示将切换资料。
- 查不到时显示“当前资料库没有找到足够依据”，不编造。
- 结果中的页码来自 chunk metadata，不由模型生成。

### 7.4 失败表现

- 查出处结果只有自然语言，没有来源卡片。
- 点击来源离开阅读器后无法返回。
- 结果页码与 PDF 实际页不一致且没有提示。
- 查不到时生成看似确定的结论。
- 阅读器内查出处和外部“找出处 / 查证据”结果格式不一致。

## 8. 与 MVP 的关系

这轮 UI 迭代应当增强以下 MVP 指标：

| MVP 指标 | UI 需求如何帮助 |
|---|---|
| 用户愿意导入真实资料 | 阅读器更像真实工具，降低“只是 demo”的感觉 |
| 用户愿意连续提问 | 打字机输出和 Markdown 降低阅读负担 |
| 用户点击来源 | 来源卡片清楚、PDF 跳转自然 |
| 用户相信回答 | 页码、原文、翻译辅助共同降低查证成本 |
| 用户复制到笔记/PPT/论文 | Markdown 结构和证据卡片更容易整理 |

不应新增的 MVP 指标：

- 是否喜欢文献图谱。
- 是否愿意多人协作。
- 是否能自动写完整论文。
- 是否能生成精美笔记。

## 9. 推荐实现顺序

### Step 1：AI Markdown 和来源展示

先让回答可读、来源可找。没有这个，后续阅读器内查出处价值会变弱。

验收：

- 3 条真实问题回答结构清楚。
- 来源卡片能看到标题、页码、片段。
- 长回答不溢出。

### Step 2：打字机输出和滚动控制

提升聊天体验，但不要牺牲稳定性。

验收：

- 用户在输出时可向上滚动。
- 回到底部后可继续跟随。
- 失败时有错误提示。

### Step 3：阅读器隐藏 chunk 列表

把阅读器变成用户能自然使用的 PDF 工具。

验收：

- chunk 列表默认不可见。
- PDF 跳页、缩放、页码输入不受影响。

### Step 4：阅读器内查出处

把 MVP 核心能力放进阅读现场。

验收：

- 阅读器内能输入观点并返回证据卡片。
- 点击证据回到 PDF 页。

### Step 5：双栏对照翻译器

最后做阅读辅助，避免它抢走“证据型问答”的核心位置。

验收：

- 选中文本或粘贴文本可翻译。
- 左原文、右中文，翻译可复制。
- 明确翻译不可替代原文引用。

## 10. 第一轮用户试用中的观察点

在真实用户试用时，重点观察：

- 用户是否更愿意读 AI 回答。
- 用户是否能从回答中快速找到来源。
- 用户在 AI 输出时是否会滚动查看上文。
- 隐藏 chunk 后，用户是否更容易专注 PDF。
- 用户是否自然使用阅读器内查出处。
- 翻译器是否帮助理解，还是让用户偏离查证据任务。

如果用户主要沉迷翻译或要求自动写全文，而不再关心来源页码，需要回到 MVP 定位提醒：第一版只验证“回答可查证”。

## 11. 不做事项

本轮明确不做：

- 完整 Zotero 替代。
- 复杂文献图谱。
- 多人协作空间。
- 自动写完整论文。
- 整篇 PDF 双语排版。
- 精美笔记系统。
- 大规模自动爬取权威资料源。

这些可以记录为后续需求，但不能影响本轮验收。
